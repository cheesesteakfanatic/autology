Trust architecture · customer holds the key

We never see your raw data.

The question stops being “do I trust their SOC 2?” and becomes “they cannot see my identifiable data.”

You run a local toolkit that anonymizes records on your own machine. You hold the only key that turns a token back into a real value. The engine computes on the anonymized input and returns token-bearing answers — which the toolkit deciphers locally, against your key, after the result has crossed back. That is a mathematical answer to a procurement question, not a promise.

The boundary

Raw stays on your machine. Only tokens cross.

A solo company can't out-attest a hyperscaler — there's no SOC 2 that beats Microsoft's. So we don't run the attestation race. We remove the question the attestation was meant to answer.

The only thing that ever crosses toward OntoForge is anonymized input. The only thing that crosses back is a token-bearing result. The key that maps tokens to identities is generated, stored, and used exclusively inside your environment. We hold no copy and have no way to request one.

  • A breach of the cloud exposes anonymized data, not PII — the blast radius is capped by construction.
  • A subpoena served on us returns tokens, not identities.
  • Per-tenant learning stays isolated — one customer's patterns never leak into another's.

Join-preserving by design

“If you scramble my data, how does the engine still find the join?”

Because the anonymization is structure-preserving, not destructive. The engine works on relationships between values — never on the literal values themselves.

same value → same token

Equality is preserved

A tail number in a registry table and in a maintenance table become the same token, so join discovery sees the value overlap exactly as it would on raw data — and the look-alike killer still fires.

cardinality · nulls · overlap

Distribution shape is preserved

Tokenization is a bijection on the value domain, so cardinality, null rate, value-set overlap, and key-uniqueness are identical over tokens. The confidence proxy is unchanged.

you choose, per column

Only the needed shape survives

You declare per column what semantic shape must cross — numeric measures stay rangeable, identifiers become opaque tokens. The toolkit guarantees that shape and nothing more.

The induced ontology, the resolved entities, the typed relationships, and the per-value provenance ledger are all computed correctly on tokens — and every citation points at a token. Provenance survives the round trip: exported AMBER estates reference tokens too, so the bundle is anonymized at rest. Hold the key and you can replay and decipher it; without the key, it's just tokens.

Auditability

We open-source the toolkit so you can audit the boundary yourself.

The whole claim is “we never see your raw data.” The only way to make that verifiable rather than asserted is to let your security team read the exact code that runs in your environment, against your key.

And the moat doesn't live in the toolkit. Our proprietary inventions — the distribution-aware confidence proxy, the execute-the-join validator, the typed-relationship ensemble, the bitemporal provenance store — are the closed core. The anonymizer is a client-side boundary tool. Opening it costs no moat and buys maximal trust.

  1. The key never leaves.
    Read the open-source toolkit; capture egress during anonymize and confirm no code path transmits the key.
  2. Only tokens cross.
    Capture egress during a full run; confirm the payload is tokens, and a known sensitive value never appears on the wire.
  3. Equality is the only default.
    Confirm tokenization is a deterministic bijection per field-class — the explicit knob trading join utility against residual risk.
  4. The result deciphers locally.
    Confirm the decipher step runs in your environment against the local key, and the returned payload is token-bearing.
  5. Export stays anonymized.
    Confirm AMBER bundles and the provenance ledger reference tokens — the estate is anonymized at rest and needs the local key.

No overselling

Demo-grade cipher today; a real KMS in production.

A trust product that hides its own limitations is not a trust product. So we state the distinction in writing.

What's true today, and what production adds

The reference toolkit uses a demo-grade, deterministic tokenization cipher. It is correct about the property that matters — same value to the same token, structure preserved, key held locally — and is sufficient to demonstrate and audit the whole boundary end to end. It is not a hardened cryptographic deployment.

A production deployment wants a real key-management system: keys generated and rotated in a customer-controlled KMS or HSM, per-field key derivation, authenticated tokenization, and an explicit threat model for the residual inference risks any deterministic scheme carries (frequency analysis on low-cardinality fields; correlation across joins). The field-class policy is where you trade join utility against those risks.

What does not change between the two is the architecture: the key lives in your environment, OntoForge computes on tokens, and the result is deciphered locally. The cipher is an implementation detail behind a boundary that is the actual product.

The full architecture — for a buyer and for an auditor — is written up in docs/ANONYMIZATION.md in the repository.
