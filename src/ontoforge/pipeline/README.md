# `ontoforge.pipeline` — the generic estate engine

OntoForge on **any** directory of data, answering over the ontology it
**induced** (whitepaper §11.3 Phase 3 — the STRATA swap-in, inverted into the
product's core capability). Point it at a directory of CSV/Parquet files and
it discovers tables, profiles them, induces an ontology, resolves cross-table
identity, materializes a bitemporal world under that induced ontology with
constraint-H provenance, and LODESTONE answers questions over it with
atom-level citations.

No frozen module is modified: STRATA's candidate evidence is read back from
its public `StrataResult` artifacts, ER runs through the M5 cascade's
name-flavored feature path, and conformance/ER enrichment happen on the
pipeline's side of the interface.

## Programmatic API (what the server/UI layers should call)

```python
from ontoforge.contracts import SpineProfile
from ontoforge.hearth import Hearth
from ontoforge.ledger import SqliteLedger
from ontoforge.lodestone import Lodestone
from ontoforge.pipeline import discover_sources, induce_estate, materialize_induced
from ontoforge.spine import DecisionSpine

estate = discover_sources("/any/dir/of/csvs")        # step 0: discovery
ledger = SqliteLedger("ledger.sqlite")
artifacts = induce_estate(estate, ledger)            # M3 profiles + M4 STRATA
hearth = Hearth("hearth_root", ledger)
stats = materialize_induced(                         # generic ER + commit
    estate, artifacts.ontology, artifacts, hearth, ledger
)
# materialize_induced ENRICHES artifacts.ontology IN PLACE (measure patches,
# unit annotations, ER link properties, grounding surface forms) — persist
# THAT object (ontoforge.vista._pipeline.save_ontology) as the ontology `ask`
# must load. The CLI saves it as <project>/ontology.materialized.json.
engine = Lodestone(artifacts.ontology, hearth, ledger,
                   DecisionSpine(SpineProfile(), model_client=None))
answer = engine.ask("How many orders are there?")    # cited or abstained
```

The aviation estate (`ontoforge.estates.load_estate()`) flows through the
same three calls unchanged — that IS the STRATA swap-in
(`ontoforge materialize --ontology induced`; see `docs/SWAPIN_REPORT.md`,
regenerated by `uv run python scripts/swapin_eval.py`).

`stats` carries `{entities, cells, links, classes, er, plans}`; `er` is the
per-class resolution summary the CLI renders, `plans` the candidate->class
materialization plans (for inspection/debugging).

## CLI

```
ontoforge init proj --source /any/dir     # generic estate (vs bundled aviation)
ontoforge ingest -p proj [--limit N]      # M1 CDC + RAW mirror (limit is sticky)
ontoforge profile -p proj                 # M3 keys/FDs/INDs/units
ontoforge induce -p proj                  # M4 STRATA -> ontology.json
ontoforge resolve -p proj                 # generic M5 ER -> resolved.json
ontoforge materialize -p proj             # induced world (constraint-H prov)
ontoforge ask "..." -p proj               # LODESTONE over the induced world
ontoforge dashboard "..." -p proj         # VISTA over the induced ontology
```

Aviation projects default to `--ontology gold` (§11.3 de-risking slice);
`--ontology induced` runs the swap-in. Generic projects only have `induced`.

## Modules (pipeline order)

| module | step | what it does |
|---|---|---|
| `discover.py` | 0 | any `*.csv`/`*.parquet` dir -> estate dict; wart-preserving string loads; M3 candidate-key detection picks key columns; content-addressed row-key fallback for keyless tables (bit-for-bit the CDC convention) |
| `induce.py` | 1 | M3 profiles + cross-table INDs -> M4 STRATA induction; keeps the full `StrataResult` (`InducedArtifacts`) |
| `mapping.py` | 2 | candidate -> admitted-class evidence recovery: one `ClassPlan` per surviving candidate (`table` / `decomp` / `hub` kinds), property->column resolution through STRATA's own property-cluster normalizer and `_class_of_candidate` (read-only import — emitter and materializer can never disagree) |
| `variants.py` | 3a | identifier-variant domain discovery: the SAME identifier written bare vs prefix-tagged across tables ("4669X" vs "N4669X"), purely data-driven (residual-overlap gain over plain overlap) |
| `er_generic.py` | 3b | cross-table identity domains per plan (key clusters + IND lhs columns + unique name-like alternates); name-like domains run the M5 cascade's "operator" (NAME) path; variant domains resolve exactly; code-like domains stay exact-match (the FK index) |
| `conform.py` | 4a | per-column conformance: null tokens, numeric coercion (>= 98% parse floor), unit conformance (declared > dominant > §3.2 magnitude heuristic), `<prop>_unit` source annotations, temporal/identifier columns pinned to strings |
| `enrich.py` | 4b | pipeline-side ontology enrichment (in place, additive only): measure patches, unit-annotation companions, ER/variant-backed link properties, grounding surface forms ("Source table: ..."), synonym suffix-phrase expansion |
| `materialize.py` | 5 | the world build: entities per row/cluster/distinct-value, ValueCells conformed, LinkCells for IND-backed + ER-resolved links + text identifier spotting; every cell's `prov_ref` is an interned term over atoms minted from the ACTUAL source cells (constraint H) |

## Gates (tests/pipeline)

* `test_generic_pipeline.py` — the product claim on a NON-aviation corpus
  (deterministic retail estate built in `tests/pipeline/conftest.py`):
  >= 4 induced classes, FK links recovered, ER folds supplier spelling
  variants (150 true suppliers), constraint-H provenance on every committed
  cell, 5/5 questions fully correct + cited over the induced world, and the
  unanswerable one abstains.
* `test_swapin_aviation.py` (slow) — the aviation competency suite over the
  induced world: >= 10/15 answerable correct (mission floor 8/15; currently
  11/15 vs gold's 15/15), 100% citation coverage, 0 confidently-wrong, both
  unanswerables abstain, trick-unit rejected by the type checker.
  Per-question induced-vs-gold analysis: `docs/SWAPIN_REPORT.md`.

Note on `--limit`: the gold competency answers are functions of the FULL
pinned fixture corpus (AMD-0006), so subsampled runs (`swapin_eval.py
--limit N`) are diagnostics only — both worlds' scorecards are depressed by
construction because the expected answers no longer hold over head-N data.
